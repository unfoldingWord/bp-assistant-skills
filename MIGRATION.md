# Migration to AWS Chainguard

Instructions for migrating workspace data and setting up the curation pipeline on the new host.

## 1. Create migration bundle (on current server)

```bash
cd /srv/bot

# Bundle hand-curated data (~30M compressed)
tar czf /tmp/workspace-migrate.tar.gz \
  -C /srv/bot/workspace \
  data/editor-feedback data/editor-notes data/editor-ult data/editor-ust data/editor-ult-english \
  data/quick-ref data/en_tw data/ta-flat \
  data/abstract_nouns.txt data/books.txt data/translation-issues.csv data/tw_headwords.json \
  output/ \
  -C /srv/bot \
  app/data/
```

### What's in the bundle

| Dir/File | Why | Size |
|----------|-----|------|
| `data/editor-*` | Human editor feedback and corrections | ~12M |
| `data/quick-ref/` | Accumulated ULT/UST rendering decisions (CSV) | ~52K |
| `data/en_tw/` | Consolidated Translation Words (manually curated from Door43) | ~4M |
| `data/ta-flat/` | Flattened Translation Academy (unknown provenance, just migrate) | ~2M |
| `data/abstract_nouns.txt` | Hand-curated word list for issue detection | static |
| `data/books.txt` | OT book code list | static |
| `data/translation-issues.csv` | All 94 issue types (master reference) | static |
| `data/tw_headwords.json` | Translation Words headword lookup | ~68K |
| `output/` | Generated ULT, UST, notes, issues, reviews (built upon) | ~14M |
| `app/data/` | Sessions, metrics, pending merges, checkpoints | ~184K |

### What's NOT in the bundle (regenerated by curation script)

- `data/published_ult/`, `data/published_ust/`, `data/published-tns/` -- fetched from Door43
- `data/published_ult_english/`, `data/published_ust_english/` -- extracted from aligned USFM
- `data/hebrew_bible/`, `data/t4t/` -- fetched from Door43
- `data/glossary/` -- fetched from Google Sheets
- `data/templates.csv`, `data/issues_resolved.txt` -- fetched from Google Sheets/Docs
- `data/cache/` -- rebuilt by index scripts
- `door43-repos/` -- shallow clones created at runtime by door43-push

## 2. Transfer to new host

```bash
scp /tmp/workspace-migrate.tar.gz user@new-aws-host:/tmp/
```

## 3. Unpack on new host

```bash
# Ensure /srv/bot exists with the git repos cloned
cd /srv/bot

# Unpack curated data
tar xzf /tmp/workspace-migrate.tar.gz

# Verify
ls workspace/data/editor-ult/
ls workspace/output/AI-ULT/
ls app/data/sessions/
```

## 4. Run initial data setup

The curation logic runs **inside the Docker container** via MCP tool or admin DM. After deploying the bot:

**Option A: Admin DM to the bot in Zulip**
```
setup data
```

**Option B: MCP tool call (from Claude during a pipeline)**
```
curate_published_data({ step: "setup" })
```

Both are equivalent to a full force-fetch. This will:
1. Discover published OT books from Door43 releases API (currently 25 books, v88)
2. Fetch aligned ULT, UST, TN, Hebrew Bible, T4T for each book
3. Fetch glossary, templates, and issues_resolved from Google Sheets/Docs
4. Extract unaligned English ULT & UST (via usfm-js proper USFM parser)
5. Resolve any missing GL quotes on TNs
6. Build Strong's index, UST index, and TN index in `data/cache/`

## 6. Verify

After setup completes, the bot will DM you a summary. You can also check via DM:

```
update data check
```

This reports the current release tag, book count, and whether any new books are available.

## 7. Ongoing updates

**Admin DM commands:**
```
update data              # Normal update (only fetches if release changed or cache stale)
update data force        # Force refetch everything
update data check        # Dry run — report what's changed
update data fetch-door43 # Just Door43 fetch step
update data build-indexes # Just rebuild indexes
```

**MCP tool** (available to Claude in pipelines):
```
curate_published_data({ step: "check" })
curate_published_data({ force: true })
curate_published_data({ step: "build-indexes" })
```

Updates should be run monthly or after a new Door43 release (releases are infrequent).

## 8. Docker considerations

- Curation runs **inside** the Docker container (CommonJS module in `app/src/curate-data.js`)
- `/workspace` is bind-mounted, so fetched data persists across container rebuilds
- `usfm-js` is an app dependency — installed during `docker compose build`, no host-side setup needed
- The standalone `.mjs` script in workspace is also available for host-side use if needed (requires host npm install)
